# Furrlytics

我想用Claude Code 做一个科学养猫的 iPhone 软件。它要集成养猫咪的各方面信息并进行智能分析和养育建议，具体包含：

## **首次启动引导流程**

创建猫咪 Profile → 设置家的位置 → 连接 Petlibro（可选）→ 进入主界面。这个流程做成线性的几步引导页。技术实现： `@AppStorage` 的布尔值标记是否完成了 onboarding，没完成就展示引导流程。
用户输入 Petlibro 账号密码后，密码经 MD5 哈希处理，凭证存入 Keychain。

## 数据采集（主要功能）

1. 进食：时间、食物种类（干粮、湿粮）。使用petlibro的API - 记录喂了多少干粮。湿粮则手动记录。
    1. Petlibro 没有官方开放的开发者API，但好消息是社区已经把它的云端API逆向工程做得相当成熟了。开发者通过分析PetLibro官方APP的网络流量，成功逆向出了关键的API端点，包括基于token的认证系统，密码需要MD5哈希后发送。 [Corti](https://corti.com/bringing-petlibro-smart-feeders-to-apple-home-building-a-homebridge-plugin/) 目前已经有完整的 Home Assistant 集成和 Homebridge 插件可供参考。
    2. 两个风险：一是 Petlibro 随时可能改接口或加验证，逆向 API 没有稳定性保障；二是 MD5 哈希密码这种方式说明他们的安全设计比较老派，但也意味着你需要在本地存储用户的 Petlibro 凭证，要做好 Keychain 加密存储。
    3. 目前不做接口挂了之后的降级（手动输入兜底）。务实的做法是：在代码里把 Petlibro 相关的逻辑做好模块化隔离（单独的 Service 层），这样万一真要改，你只需要动那一个文件，不会牵连其他功能。
2. 零食（照片/链接）：记录猫咪喜欢的零食，方便主人下次购买。上传照片或者产品链接 → 拍照后OCR识别产品名称 → 用户做确认/编辑 → 归档。
    1. Apple Vision Framework 的文字识别对中英文都支持，可以直接用 `VNRecognizeTextRequest` 在设备端完成 OCR，不需要第三方服务。
3. 训练里程碑：记录训练小猫的进度。进度做成"里程碑"式记录而不是每日打卡。
   
    比如：用户点击"记录里程碑" → 输入里程碑名称（如"第一次成功击掌"）→ 可选择从相册上传或现场拍摄短视频 → 保存
    
    储存方式：限制为照片 + 短文字描述，照片直接存缩略图到 app 沙盒；视频存 PHAsset 引用相册（避免复制大文件）。如果用户从相册删了那个视频，存的引用就指向空了，那么在展示时做一个"视频已被删除"的兜底状态。
    
4. 陪伴猫咪玩耍：次数、时间。考虑到每次和猫咪陪玩可能都是临时起意，所以使用Siri Shortcuts + 每晚手动输入兜底。需要注册 Shortcut Intent。
5. 离家时间（手机定位）：接入手机位置，每次离开家之后都记录离开家的时间。如果是长期旅游（>1天），则注明主人这几天没有回家。
    1. 较精准的方案是用 `CLCircularRegion` 做地理围栏（以家为圆心设定半径），这比 Significant Location Change 更准确，功耗也可控。
    2. 实际使用建议：半径设 150-250 米（太小容易因 GPS 漂移误触发，太大会延迟检测），加一个时间防抖（离开围栏后等 5-10 分钟再标记为"离家"，避免下楼拿个快递就触发）。
6. 带猫咪出门的时间：手动记录。可以在检测到用户离家时弹出提示"这次带猫咪了吗？"。如果用户没有点击“带猫咪了”，那默认用户没有带猫咪，进入离家时间的逻辑。但是防止用户忽略这条通知，需要确保用户后续可以手动修正——比如回家后在时间线里看到一条"离家记录"，可以点进去补标"这次带了猫"。
7. 驱虫提醒 - 让用户设置不同驱虫药物的时间间隔。比如内驱药一个月一次，外驱药两个月一次。首次设置后，用户每一次记录了驱虫后自动安排下一次提醒（比如记录"今天做了内驱" → 自动在30天后推送提醒）。实现上就是每次记录驱虫事件时用 `UNNotificationRequest` 安排一条本地定时通知。
8. 多猫逻辑。初次进入app时询问猫咪数量和名字，给猫咪创建profile：MVP只存ID, 名字, 照片。后续预留字段：年龄、体重、品种。如果只有一只猫咪，则以上所有数据都默认是当前猫咪，不需要询问猫咪名字。如果有多只猫咪，则以上数据录入都询问猫咪名字。

### 家的设置

**地理围栏需要用户手动设"家"的位置。** 需要一个设置流程让用户标定自己家的坐标。最简单的方式是"你现在在家吗？点击设为家的位置"，用当前 GPS 坐标。但考虑 GPS 在室内精度不够好，所以给用户一个微调的地图界面——展示一个地图，中心是当前位置，用户可以拖动调整。

**搬家了怎么办。** 地理围栏的"家"坐标，在用户搬家后需要重新设置。在设置页里放一个"更新家的位置"入口。

### 「记录疲劳」的风险分析

进食只需记录湿粮，可能两三天喂食记录一次，高频。零食：约2周1次，频率低。训练：频率更低。陪伴猫咪玩耍：用了Siri之后会方便很多，手动兜底1次/天，高频。离家时间：不用手动记录。带猫咪出门的时间：只需要点击弹窗确认，无需手动点开。多猫：影响了进食、零食、陪伴猫咪玩耍的逻辑。综上，手动记录的高频维度共进食和玩耍两个，能控制在一天2-3次手动记录。

因为高频手动记录就两个维度，那在交互设计上给它们做「快速通道」——比如 app 首页直接露出"记录进食"和"记录玩耍"两个大按钮，点一下就能完成记录（选猫→选食物/输入时长→完成，其中选猫这一步如果是单猫家庭可以跳过），争取 3 秒内完成一次记录。往下滑是数据回显，其他低频功能收到二级页面。

### **编辑和删除记录**

用户一定会录错——选错了猫、时间记反了、手滑点了提交；或者忽略了离开家是否带上猫咪的提醒。需要在时间线里支持：点击某条记录进行编辑或删除，以及离开家是否带上猫咪的修改。SwiftUI 里 List 的 swipe-to-delete 是现成的，编辑就是点击后弹出和新建相同的表单、预填已有数据。

### 数据存储架构

MVP 用 **SwiftData 同步**。不需要后端和自建服务器。轮询方案是：APP 在前台时或后台定时刷新时，主动调用 Petlibro API 拉取最新数据然后存本地。能接受数据有几小时的延迟。

数据模型如何设计？思路：

```python
Event {
    id, timestamp, type, source, catId, 
    metadata（JSON，不同事件类型存不同字段）
}
```

type 可以是 feeding / treat / play / training / away / outing / deworming 等。source 标记数据来源（manual / petlibro / location）。这样 AI 分析层可以统一查询所有事件做关联分析，比如"主人离家时间长的日子，猫叫声频率是否增加"。

在 SwiftData 里可以用 Swift 的枚举 + associated values 来做类型安全的 metadata，兼顾灵活性和安全性：

```swift
enum EventType: Codable {
    case feeding(FeedingDetail)
    case play(PlayDetail)
    case away(AwayDetail)
    // ...
}
```

**SwiftData schema migration 的策略：**MVP 阶段将持续测试App一个月，在MVP阶段，schema不用加版本号、数据模型不稳定时接受删库重装。

## 可视化数据回显

以下是具体实现方式：

1. **时间线视图**
   
    一个按时间倒序排列的事件列表，类似微信朋友圈或 iOS 健康 app 的记录流。每条记录是一个卡片，显示时间、事件类型图标、简要内容（比如"🍗 湿粮 · 鸡肉罐头" 或 "🎾 玩耍 · 15分钟"）。SwiftUI 里就是一个 `List` 或 `ScrollView` + `ForEach`，按日期分组，非常基础的实现。
    
2. 基础的**日报周报卡片**。
   
    在前端代码里计算后显示一些固定的卡片，如：米米本周吃了3次湿粮，玩耍了5次，你每天离家平均6小时。
    用 SwiftData 的 `@Query` 或者写几个查询方法，比如"过去7天 type == .feeding 的事件数量""过去7天 type == .play 的事件总时长""过去7天 type == .away 的事件平均时长"，然后把结果填进几个预设的卡片模板里渲染。
    卡片可以放在首页快速操作下面。这样首页的信息架构就是：快速操作（记录湿粮/记录玩耍）→ 周报摘要 → 最近事件时间线。
    

**时区和日期边界**：用用户设备时区。一天的分界线设置为4:00 AM。
理解：用户录入时看到的永远是真实时间，比如凌晨1点记录了一次喂食，显示的就是"1:00 AM"。但在周报卡片、时间线按天分组这些需要判断"这条记录属于哪一天"的地方，凌晨1点的记录会被归到前一天。
实现上就是在做聚合查询时，把事件的时间戳减去4小时再取日期。比如4月3日凌晨1点的事件，减4小时变成4月2日21点，取日期就是4月2日。4月3日早上8点的事件，减4小时变成4月3日4点，取日期还是4月3日。这个逻辑只需要写一个工具函数，所有统计查询都调用它来确定"逻辑日期"，不影响存储和展示的真实时间。

## 后续升级

以下功能MVP暂不考虑，但在此处注明，以便后续升级。

1. 陪伴猫玩耍：App intent shortcut - app 内提供一键添加 Shortcut 的引导。
2. 可视化数据回显：时间线长之后，增加日历视图
3. 后续考虑接入apple watch进行简单的手动记录，如：进食、陪伴猫咪玩耍的计时功能、带猫出门确认按钮
4. 暂不考虑用户注册和多人家庭。后续考虑：需要支持多用户，只有所有家庭成员都离开时才标记"猫独处"。后续如果能有用户注册和绑定家庭的功能，则可以增加这个逻辑。
5. 数据储存架构：多人家庭功能出现后，加入后端和 **iCloud CloudKit。** Apple 提供免费的 CloudKit 额度对个人项目够用，而且后续做家庭共享时 CloudKit 原生支持数据共享。
6. 我计划在实现以上数据收集的情况下，后期用**AI 生成分析和建议** - 
    1. 使用规则引擎（e.g. if 进食量连续3天下降 then 提醒; if 离开家很久 then 提醒主人回家看看）
    2.  LLM做总结（把猫的数据喂给 Claude/GPT 做自然语言分析）
    
    在MVP阶段也暂时不实现，但是需要考虑到后续迭代：逻辑如何结合各类数据生成、LLM在哪里跑 等。
    
7. 不考虑接入furrytail（自动猫砂盆）。暂时不考虑监测猫咪屎尿。后续：健康模块的建立。考虑加入体重追踪（定期称重）、便便记录（形态/频率）、呕吐记录等。
8. 不考虑猫咪运动/叫声的监测。因为需要**摄像头硬件**和集成软件数据的接入，所以最复杂，在MVP阶段舍弃。后续考虑：
    1. 运动：摄像头检测到运动 → 记录活动时段，不需要考虑猫咪行为分类。猫咪叫声有现成的分析API。
    2. 声音：Apple 的 Sound Classification API（`SNClassifySoundRequest`）原生就支持识别猫叫声，而且跑在设备端，不需要上传音频。但问题是：这个检测是跑在手机上还是摄像头端？如果用户不在家，手机不在现场，就需要摄像头端有声音事件检测能力。
    3. 难点可能在于摄像头或**后端**的数据储存和实时分析/传输的实现。
9. **用户群体增大**后，给Petlibro的API 做一个兜底的优雅降级 — API挂掉之后让用户能够手动输入进食。可能涉及截图分析。
